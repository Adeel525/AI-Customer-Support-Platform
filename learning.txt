SECTION 1: PROJECT OVERVIEW (ELEVATOR PITCH)
================================================================================

WHAT IS THIS PROJECT?
---------------------
A full-stack SaaS platform that lets companies:
  - Sign up and create a workspace (multi-tenant)
  - Upload documents and crawl websites to build a knowledge base
  - Train an AI chatbot using RAG (Retrieval-Augmented Generation)
  - Embed the chatbot on any website via a JavaScript widget
  - Handle customer conversations with AI + human escalation
  - Auto-generate support tickets when AI confidence is low
  - View analytics (CSAT, ticket volume, top questions, accuracy trends)

WHO IS IT FOR?
--------------
B2B SaaS — companies that repeatedly answer the same customer questions
(pricing, refunds, product info, installation, troubleshooting).

WHAT PROBLEM DOES IT SOLVE?
---------------------------
Companies waste time answering repetitive support questions. This platform
automates 70%+ of support using RAG — the AI retrieves real company data
before generating answers, reducing hallucinations and improving accuracy.

BACKEND EVOLUTION
-----------------
Originally built with FastAPI; migrated to Django REST Framework while keeping:
  - Same /api/v1/ URL contract (frontend unchanged)
  - MongoDB as primary datastore (via mongoengine)
  - Same RAG / Celery / Pinecone architecture
  FastAPI source kept for reference at: backend/app_fastapi_backup/

REQUEST FLOW EXAMPLE (Send Chat Message)
-----------------------------------------

  1. Widget POST /api/v1/public/chat/{chatbot_id}/message
  2. PublicChatMessageView (AllowAny)
  3. Load Chatbot by id where is_active=True
  4. Create/find Conversation in MongoDB (mongoengine)
  5. Save user Message
  6. RAGEngine(workspace_id).query() — embed, search, rerank, LLM generate
  7. Save assistant Message with confidence + sources
  8. If confidence < 0.6 → auto-create Ticket
  9. Return JSON: { content, confidence, sources, conversation_id }


WHY DJANGO REST FRAMEWORK?
--------------------------
  - Mature ecosystem (admin, auth patterns, middleware, CORS packages)
  - Clear View / Serializer / Permission separation
  - Easy Celery integration via django.conf settings
  - Familiar for enterprise Python teams
  - Still uses MongoDB for flexible SaaS documents (mongoengine ODM)
  - SQLite only for Django's built-in admin/session tables

SECTION 5: CORE CONCEPTS & TECHNIQUES USED
================================================================================

1. RETRIEVAL-AUGMENTED GENERATION (RAG)
--------------------------------------
   Instead of relying only on the LLM's training data, RAG:
     a. Retrieves relevant document chunks from your knowledge base
     b. Injects them as context into the LLM prompt
     c. Generates an answer grounded in your actual company data

   Why RAG over fine-tuning?
     - No retraining needed when docs change
     - Source citations possible
     - Lower cost
     - Works with proprietary/private data immediately

2. HYBRID SEARCH
----------------
   Combines two retrieval methods:
     - Vector search (semantic similarity via Pinecone embeddings)
     - Keyword search (MongoDB / mongoengine content__icontains)
   Better recall than either method alone.

3. QUERY REWRITING
------------------
   Follow-up questions like "What about refunds?" are rewritten by the LLM
   into standalone queries using conversation history, improving retrieval
   accuracy for multi-turn conversations.

4. CONFIDENCE SCORING
---------------------
   Two signals combined:
     - Average embedding similarity score from retrieved chunks
     - LLM self-assessment (asks model to rate 0.0–1.0)
   If confidence < 0.6 → auto-escalate to human agent + create ticket.

5. MULTI-TENANCY
-----------------
   Each company (workspace) has completely isolated data:
     - MongoDB: every document has workspace_id field
     - Pinecone: separate namespace per workspace (workspace_{id})
     - JWT: workspace_id claim in access token
     - Header: X-Workspace-Id on every authenticated request
     - require_member() validates membership before workspace operations

6. CHUNKING STRATEGY
--------------------
   Documents split into overlapping chunks for better retrieval:
     - Chunk size: 800 characters (~500-1000 tokens)
     - Overlap: 150 characters (preserves context across boundaries)
     - Tool: LangChain RecursiveCharacterTextSplitter
     - Separators: paragraph → line → sentence → word

7. EMBEDDINGS
-------------
   Text converted to 1536-dimensional vectors for semantic search:
     - Model: OpenAI text-embedding-3-small
     - Stored in: Pinecone (production) or in-memory dict (dev fallback)
     - Similar vectors = semantically similar content

8. VECTOR DATABASE (Pinecone)
------------------------------
   Specialized DB for storing and searching high-dimensional vectors:
     - Index: support-kb
     - Metric: cosine similarity
     - Namespace per tenant for isolation
     - Top-K retrieval (K=20, reranked to top 5)

9. BACKGROUND JOB PROCESSING (Celery)
--------------------------------------
   Long-running tasks run asynchronously so API responds fast:
     - Document ingestion (extract → chunk → embed → store)
     - Website crawling
     - Nightly analytics aggregation
     - Conversation summarization
   Celery app: celery -A config worker / beat
   Tasks live in: api/tasks.py

10. ROLE-BASED ACCESS CONTROL (RBAC)
------------------------------------
    4 roles with granular permissions:
      owner   → all permissions
      admin   → all except billing
      agent   → tickets, live chat, analytics view
      viewer  → analytics view only
    Enforced via DRF permission classes:
      HasWorkspacePermission(Permission.MANAGE_KNOWLEDGE.value)

11. JWT AUTHENTICATION (Custom for mongoengine Users)
------------------------------------------------------
    Django's default User model is NOT used for app users.
    Custom JWTAuthentication loads mongoengine User by token sub claim.
    Access token  → 30 minutes
    Refresh token → 7 days
    Claims: user_id (sub), workspace_id, role, type (access|refresh)
    Files: api/jwt_utils.py, api/authentication.py

12. DRF PERMISSIONS & AUTHENTICATION CLASSES
---------------------------------------------
    Instead of FastAPI Depends(), Django uses:
      authentication_classes = [JWTAuthentication]
      permission_classes = [IsAuthenticatedJWT, HasWorkspacePermission(...)]
    Tenant helpers: get_workspace_id(request), require_member(user_id, ws_id)

13. MONGOENGINE ODM (Not Django ORM for app data)
-------------------------------------------------
    App collections are mongoengine Document classes.
    Django ORM/SQLite is only for admin, sessions, contenttypes.
    Connection established in api.apps.ApiConfig.ready().

14. STRATEGY PATTERN (AI Fallbacks)
------------------------------------
    LLMClient:     OpenAI → Claude → Demo message
    VectorStore:   Pinecone → In-memory dict
    Embeddings:    OpenAI → Hash-based deterministic vectors

15. CORS & APPEND_SLASH
-----------------------
    django-cors-headers for frontend origin whitelist.
    APPEND_SLASH = False so frontend paths without trailing slash work
    (e.g. /api/v1/auth/signup).

16. GRACEFUL DEGRADATION
-------------------------
    Platform works without OpenAI/Pinecone API keys:
      - Hash-based embeddings (dev mode)
      - In-memory vector store
      - Demo LLM responses
    Document upload tries Celery first, falls back to sync processing.

SECTION 7: RAG PIPELINE — DEEP DIVE
================================================================================

DOCUMENT INGESTION
------------------
  Upload (DRF multipart) → StorageService (S3/local)
  → Document(status=pending)
  → Celery process_document OR sync fallback
  → extract_document → split_text → EmbeddingService
  → VectorStore.upsert (Pinecone namespace) + DocumentChunk
  → Document(status=indexed)

QUERY PIPELINE
--------------
  Cache (Redis) → Query rewrite → Embed query
  → Pinecone top 20 + mongoengine keyword top 5
  → Rerank/dedupe top 5 → Build context
  → LLM generate → Confidence score
  → Escalate if < 0.6 → Return + cache

Key file: backend/core/ai/rag_engine.py

SECTION 18: INTERVIEW QUESTIONS & ANSWERS
================================================================================

Q1: What is this project?
A:  SupportAI — multi-tenant SaaS for AI support chatbots trained on a
    company's docs/website via RAG, with ticket auto-escalation and analytics.

Q2: Why RAG instead of fine-tuning?
A:  Instant knowledge updates, citations, lower cost, private data, per-tenant
    isolation without retraining models.

Q3: Explain the RAG pipeline.
A:  Ingest: upload → store → extract → chunk → embed → Pinecone + MongoDB.
    Query: rewrite → embed → hybrid search → rerank → LLM → confidence →
    escalate if < 0.6 → return with sources.

Q4: How is multi-tenancy implemented?
A:  (1) JWT / X-Workspace-Id + membership check
    (2) workspace_id on every mongoengine query
    (3) Pinecone namespace per workspace

Q5: Why Django REST Framework (not FastAPI)?
A:  We migrated to DRF for a structured View/Serializer/Permission model,
    mature Django ecosystem (CORS, Celery integration, admin), and team
    familiarity — while keeping MongoDB via mongoengine and the same /api/v1
    contract so the Next.js frontend did not need rewriting.

Q6: Why mongoengine instead of Django ORM?
A:  Core SaaS data is document-oriented (flexible fields for branding,
    chatbot config, analytics metrics). mongoengine maps naturally to MongoDB.
    Django ORM/SQLite is only used for admin and session tables.

Q7: How does authentication work in this Django app?
A:  Custom JWTAuthentication (not Django session auth for API). Tokens issued
    with PyJWT; User documents live in MongoDB. On each request, Bearer token
    is decoded, User loaded, membership checked via X-Workspace-Id.

Q8: How is RBAC enforced?
A:  core/enums.ROLE_PERMISSIONS maps roles to permissions.
    Views use HasWorkspacePermission("manage_knowledge") etc.
    Membership role comes from WorkspaceMember, not only the JWT claim.

Q9: What happens on low AI confidence?
A:  Conversation → escalated; Ticket auto-created with AI summary/intent/
    priority; customer is told a human will help.

Q10: How are documents processed?
A:  Upload creates Document(pending). Celery process_document (or sync
     fallback) extracts, chunks, embeds, upserts Pinecone, saves chunks,
     sets status indexed/failed.

Q11: What is hybrid search?
A:  Vector similarity (meaning) + keyword match (exact terms) merged and
     reranked for better recall.

Q12: How is the backend structured?
A:  Django project (config/) + DRF app (api/) + shared core (AI/utils).
     Views → serializers/permissions → mongoengine models / RAG services →
     Celery tasks for heavy work.

Q13: Design patterns?
A:  DRF layered architecture, custom auth/permissions, ODM models,
     strategy fallbacks for AI providers, Celery task queue, multi-tenant
     request helpers.

Q14: How does the crawler work?
A:  URL → WebsiteCrawler (httpx + BeautifulSoup) → ingest_text per page →
     same knowledge pipeline; Beat re-syncs daily.

Q15: Frontend state management?
A:  Zustand (auth), React Query (server data), Axios interceptors (JWT refresh).

Q16: How would you scale?
A:  Multiple Gunicorn/Django workers behind Nginx; MongoDB Atlas shard by
     workspace_id; Pinecone serverless; more Celery workers; Redis Cluster;
     S3 + CDN for widget.

Q17: Limitations?
A:  Stripe/voice/WhatsApp/Slack stubs; API-key auth not on all routes;
     widget missing some UX features; WebSocket live chat partial.

Q18: FastAPI vs Django in this project?
A:  FastAPI was the original async API. Migration to Django/DRF kept features
     and URLs; AI layer was converted to sync for simpler DRF views; backup
     remains in app_fastapi_backup for comparison in interviews.
