SECTION 1: PROJECT OVERVIEW (ELEVATOR PITCH)
================================================================================

WHAT IS THIS PROJECT?
---------------------
A full-stack SaaS platform that lets companies:
  - Sign up and create a workspace (multi-tenant)
  - Upload documents and crawl websites to build a knowledge base
  - Train an AI chatbot using RAG (Retrieval-Augmented Generation)
  - Embed the chatbot on any website via a JavaScript widget
  - Handle customer conversations with AI + human escalation
  - Auto-generate support tickets when AI confidence is low
  - View analytics (CSAT, ticket volume, top questions, accuracy trends)

WHO IS IT FOR?
--------------
B2B SaaS — companies that repeatedly answer the same customer questions
(pricing, refunds, product info, installation, troubleshooting).

WHAT PROBLEM DOES IT SOLVE?
---------------------------
Companies waste time answering repetitive support questions. This platform
automates 70%+ of support using RAG — the AI retrieves real company data
before generating answers, reducing hallucinations and improving accuracy.

REQUEST FLOW EXAMPLE (Send Chat Message)
-----------------------------------------

  1. Widget POST /api/v1/public/chat/{chatbot_id}/message
  2. Rate limiter checks IP (30 requests/minute)
  3. ChatbotRepository.get_public() — validate chatbot exists
  4. ChatService.send_message()
       a. Create/find conversation in MongoDB
       b. Save user message
       c. RAGEngine.query() — embed, search, rerank, LLM generate
       d. Save assistant message with confidence + sources
       e. If confidence < 0.6 → TicketService.auto_create_from_conversation()
  5. Return JSON: { content, confidence, sources, conversation_id }

SECTION 5: CORE CONCEPTS & TECHNIQUES USED
================================================================================

1. RETRIEVAL-AUGMENTED GENERATION (RAG)
--------------------------------------
   Instead of relying only on the LLM's training data, RAG:
     a. Retrieves relevant document chunks from your knowledge base
     b. Injects them as context into the LLM prompt
     c. Generates an answer grounded in your actual company data

   Why RAG over fine-tuning?
     - No retraining needed when docs change
     - Source citations possible
     - Lower cost
     - Works with proprietary/private data immediately

2. HYBRID SEARCH
----------------
   Combines two retrieval methods:
     - Vector search (semantic similarity via Pinecone embeddings)
     - Keyword search (MongoDB regex on chunk content)
   Better recall than either method alone.

3. QUERY REWRITING
------------------
   Follow-up questions like "What about refunds?" are rewritten by the LLM
   into standalone queries using conversation history, improving retrieval
   accuracy for multi-turn conversations.

4. CONFIDENCE SCORING
---------------------
   Two signals combined:
     - Average embedding similarity score from retrieved chunks
     - LLM self-assessment (asks model to rate 0.0–1.0)
   If confidence < 0.6 → auto-escalate to human agent + create ticket.

5. MULTI-TENANCY
-----------------
   Each company (workspace) has completely isolated data:
     - MongoDB: every document has workspace_id field
     - Pinecone: separate namespace per workspace (workspace_{id})
     - JWT: workspace_id claim in access token
     - Header: X-Workspace-Id on every authenticated request

6. CHUNKING STRATEGY
--------------------
   Documents split into overlapping chunks for better retrieval:
     - Chunk size: 800 characters (~500-1000 tokens)
     - Overlap: 150 characters (preserves context across boundaries)
     - Tool: LangChain RecursiveCharacterTextSplitter
     - Separators: paragraph → line → sentence → word

7. EMBEDDINGS
-------------
   Text converted to 1536-dimensional vectors for semantic search:
     - Model: OpenAI text-embedding-3-small
     - Stored in: Pinecone (production) or in-memory dict (dev fallback)
     - Similar vectors = semantically similar content

8. VECTOR DATABASE (Pinecone)
------------------------------
   Specialized DB for storing and searching high-dimensional vectors:
     - Index: support-kb
     - Metric: cosine similarity
     - Namespace per tenant for isolation
     - Top-K retrieval (K=20, reranked to top 5)

9. BACKGROUND JOB PROCESSING (Celery)
--------------------------------------
   Long-running tasks run asynchronously so API responds fast:
     - Document ingestion (extract → chunk → embed → store)
     - Website crawling
     - Nightly analytics aggregation
     - Conversation summarization

10. ROLE-BASED ACCESS CONTROL (RBAC)
------------------------------------
    4 roles with granular permissions:
      owner   → all permissions
      admin   → all except billing
      agent   → tickets, live chat, analytics view
      viewer  → analytics view only

11. JWT AUTHENTICATION
----------------------
    Stateless auth using JSON Web Tokens:
      Access token  → 30 minutes (used on every API call)
      Refresh token → 7 days (used to get new access token)
    Claims: user_id (sub), workspace_id, role, token type

12. DEPENDENCY INJECTION (FastAPI Depends)
-------------------------------------------
    Services and repositories injected via FastAPI's Depends():
      def get_service(tenant: Tenant) -> KnowledgeService:
          return KnowledgeService(get_database(), tenant.workspace_id)

13. REPOSITORY PATTERN
-----------------------
    All database access goes through repository classes.
    BaseRepository provides: create, get_by_id, update, delete, list
    with automatic workspace_id filtering and timestamp management.

14. STRATEGY PATTERN (AI Fallbacks)
------------------------------------
    LLMClient:     OpenAI → Claude → Demo message
    VectorStore:   Pinecone → In-memory dict
    Embeddings:    OpenAI → Hash-based deterministic vectors

15. AUDIT LOGGING
-----------------
    Middleware logs all POST/PUT/PATCH/DELETE operations to audit_logs
    collection with: user_id, workspace_id, action, duration, status code.

16. RATE LIMITING
-----------------
    SlowAPI limits public widget chat endpoint to 30 requests/minute per IP
    to prevent abuse of the unauthenticated public API.

17. GRACEFUL DEGRADATION
-------------------------
    Platform works without OpenAI/Pinecone API keys:
      - Hash-based embeddings (dev mode)
      - In-memory vector store
      - Demo LLM responses
    Allows development and demos without paid API keys.

SECTION 7: RAG PIPELINE — DEEP DIVE
================================================================================

DOCUMENT INGESTION PIPELINE
----------------------------

  Step 1: UPLOAD
    User uploads file via dashboard or API
    POST /api/v1/knowledge/documents/upload
    Allowed: .pdf .docx .doc .txt .md .csv .html .htm

  Step 2: STORE
    File saved to AWS S3 or local ./uploads directory
    MongoDB document record created with status: "pending"

  Step 3: QUEUE (Celery)
    process_document.delay(document_id, workspace_id)
    API returns immediately — processing happens in background

  Step 4: EXTRACT TEXT
    PDF    → PyPDF + PDFPlumber fallback
    DOCX   → python-docx
    HTML   → BeautifulSoup (removes nav/footer/script tags)
    TXT/MD → direct UTF-8 decode
    Text cleaned: whitespace normalized, excessive newlines removed

  Step 5: CHUNK
    LangChain RecursiveCharacterTextSplitter
    chunk_size=800, chunk_overlap=150
    Each chunk gets: chunk_id, page_number, source, token_count

  Step 6: EMBED
    OpenAI text-embedding-3-small → 1536-dim vector per chunk
    Fallback: SHA-256 hash → deterministic pseudo-vector (dev mode)

  Step 7: STORE VECTORS
    Pinecone upsert in namespace: workspace_{workspace_id}
    Metadata: content (first 1000 chars), document_id, page_number, source

  Step 8: STORE METADATA
    MongoDB document_chunks collection
    Document status updated to: "indexed"


QUERY / ANSWER PIPELINE
------------------------

  Step 1: CACHE CHECK
    Redis key: rag:{md5(workspace_id + question)}
    TTL: 3600 seconds (1 hour)
    Return cached result if hit

  Step 2: QUERY REWRITING
    If conversation history exists:
      LLM rewrites follow-up into standalone search query
    Example: "What about refunds?" → "What is the refund policy?"

  Step 3: EMBED QUERY
    Same embedding model as ingestion
    Query → 1536-dim vector

  Step 4: VECTOR SEARCH
    Pinecone query: top 20 results (RAG_TOP_K=20)
    Filtered by workspace namespace

  Step 5: KEYWORD SEARCH
    MongoDB regex search on document_chunks.content
    Top 5 additional results (score: 0.5 fixed)

  Step 6: RERANK & DEDUPLICATE
    Sort all results by score descending
    Remove duplicate content (first 100 chars match)
    Keep top 5 (RAG_RERANK_TOP_K=5)

  Step 7: BUILD CONTEXT WINDOW
    Format: [Source N: filename, page X]\n{chunk content}
    All 5 chunks joined with --- separators
    Injected into SUPPORT_SYSTEM_PROMPT

  Step 8: LLM GENERATION
    System prompt: company name, tone, personality, context, rules
    Messages: system + last 6 history messages + user question
    Model: GPT-4o (primary) → Claude (fallback) → demo message

  Step 9: CONFIDENCE SCORING
    Signal 1: Average retrieval similarity score
    Signal 2: LLM self-rating (0.0–1.0)
    Final confidence = max(LLM score, avg retrieval score)

  Step 10: AUTO-ESCALATION
    If confidence < CONFIDENCE_THRESHOLD (0.6):
      - Conversation status → "escalated"
      - TicketService.auto_create_from_conversation()
      - AI generates ticket title, summary, intent, priority

  Step 11: RETURN & CACHE
    Response: { content, confidence, sources[], should_escalate }
    Cache result in Redis for 1 hour


PROMPT TEMPLATE (SUPPORT_SYSTEM_PROMPT)
----------------------------------------
Rules enforced in prompt:
  - Answer ONLY from provided context
  - Say "I don't know" if context insufficient
  - Always cite sources
  - Never hallucinate
  - Suggest human agent for billing/account issues

SECTION 8: MULTI-TENANT ARCHITECTURE
================================================================================

WHAT IS MULTI-TENANCY?
-----------------------
One application instance serves thousands of companies (tenants).
Each company's data must be completely isolated from others.

HOW ISOLATION IS ENFORCED (3 LAYERS)
------------------------------------

  Layer 1 — REQUEST CONTEXT
    Every authenticated request carries workspace_id via:
      - JWT claim: workspace_id in access token
      - HTTP header: X-Workspace-Id
    get_tenant_context() in deps.py validates:
      - User is authenticated
      - User is a member of the requested workspace
    Result: TenantContext(workspace_id, user_id, role) on request.state

  Layer 2 — REPOSITORY FILTER
    BaseRepository._base_filter() automatically adds:
      { "workspace_id": self.workspace_id }
    to every MongoDB query when repository is workspace-scoped.
    No unscoped reads possible through repository layer.

  Layer 3 — VECTOR NAMESPACE
    Pinecone namespace: workspace_{workspace_id}
    Vectors from Company A can never appear in Company B's search results.

DATA MODEL
----------
  users            → global (no workspace_id)
  workspaces       → one per company
  workspace_members → links users to workspaces with roles
  ALL other collections → have workspace_id field

SECTION 9: AUTHENTICATION & AUTHORIZATION
================================================================================

SIGNUP FLOW
-----------
  1. POST /auth/signup { email, password, full_name, workspace_name }
  2. Create user document (password hashed with bcrypt)
  3. Create workspace document (auto-generated slug)
  4. Create workspace_member (role: owner)
  5. Send verification email (or auto-verify if no SMTP)
  6. Return JWT access + refresh tokens

LOGIN FLOW
----------
  1. POST /auth/login { email, password }
  2. Verify bcrypt password hash
  3. Find user's workspace memberships
  4. Generate tokens with workspace_id + role claims
  5. Return tokens

TOKEN REFRESH FLOW
------------------
  1. Access token expires (30 min)
  2. Frontend Axios interceptor catches 401
  3. POST /auth/refresh { refresh_token }
  4. New access + refresh tokens issued
  5. Original request retried automatically

GOOGLE OAUTH FLOW
-----------------
  1. GET /auth/google/login → redirect to Google
  2. User authenticates with Google
  3. GET /auth/google/callback → Authlib processes token
  4. handle_oauth_user() — create or find user
  5. Redirect to frontend: /auth/callback?access_token=...&refresh_token=...
  6. Frontend stores tokens, calls /auth/me, redirects to dashboard

SECTION 18: INTERVIEW QUESTIONS & ANSWERS
================================================================================

Q1: What is this project and what problem does it solve?
A:  SupportAI is a multi-tenant SaaS platform that lets companies build
    AI customer support chatbots trained on their own documents and websites.
    It solves the problem of companies repeatedly answering the same support
    questions by using RAG to retrieve company-specific information before
    generating AI responses, reducing support workload by up to 70%.

Q2: Why did you choose RAG instead of fine-tuning an LLM?
A:  RAG is better for this use case because:
    (1) Knowledge updates instantly when documents change — no retraining
    (2) Source citations are possible — reduces hallucination risk
    (3) Lower cost — no GPU training required
    (4) Works with proprietary data without exposing it to model training
    (5) Each tenant has isolated knowledge — fine-tuning can't do that easily

Q3: Explain the RAG pipeline step by step.
A:  Ingestion: Upload → S3 → Extract text → Chunk (800/150 overlap) →
    Embed (OpenAI 1536-dim) → Store in Pinecone + MongoDB.
    Query: Cache check → Query rewrite → Embed query → Vector search (top 20)
    + Keyword search (top 5) → Rerank to top 5 → Build context → LLM generate
    → Confidence score → Auto-escalate if < 0.6 → Return with sources.

Q4: What is multi-tenancy and how did you implement it?
A:  Multi-tenancy means one app serves many companies with isolated data.
    We enforce isolation at 3 levels:
    (1) JWT/header carries workspace_id, validated against workspace_members
    (2) BaseRepository auto-filters all queries by workspace_id
    (3) Pinecone uses separate namespace per workspace

Q5: What tech stack did you use and why?
A:  FastAPI for async Python backend with auto-generated OpenAPI docs.
    Next.js 15 for SSR/SSG React frontend with App Router.
    MongoDB for flexible document schema (schemaless, good for SaaS).
    Pinecone for managed vector search (no ops overhead).
    Celery + Redis for async document processing.
    TypeScript throughout frontend for type safety.

Q6: How does authentication work?
A:  JWT-based stateless auth. On login, server issues access token (30 min)
    and refresh token (7 days). Frontend stores in localStorage. Axios
    interceptor adds Bearer token to every request. On 401, automatically
    calls /auth/refresh to get new tokens. Google OAuth via Authlib redirects
    to Google, callback creates/finds user, redirects to frontend with tokens.

Q7: Explain the RBAC system.
A:  4 roles: owner, admin, agent, viewer. 9 permissions mapped to roles
    in ROLE_PERMISSIONS dict in models/enums.py. Endpoints use
    require_permission(Permission.X) FastAPI dependency which checks
    the user's role from workspace_members (not just JWT claim) and
    raises 403 if permission missing.

Q8: How does the chatbot widget work without authentication?
A:  Widget calls public API endpoints (/public/chat/{chatbot_id}/message)
    with no auth token. Security via: (1) rate limiting 30 req/min per IP,
    (2) chatbot must be is_active=true, (3) chatbot_id is UUID (not guessable),
    (4) customer_id generated client-side and stored in localStorage.

Q9: What happens when the AI is not confident in its answer?
A:  If confidence score < 0.6 (CONFIDENCE_THRESHOLD):
    (1) Conversation status changes to "escalated"
    (2) TicketService.auto_create_from_conversation() is called
    (3) LLM generates ticket: title, summary, intent, priority, category
    (4) Customer sees message that a human agent will help them

Q10: How are documents processed after upload?
A:  Upload returns immediately with status "pending". Celery task
     process_document runs in background: downloads from S3 → extracts text
     (PDF/DOCX/HTML/TXT) → chunks with LangChain → embeds each chunk →
     upserts to Pinecone → saves metadata to MongoDB → updates status to
     "indexed" or "failed" with error message.

Q11: What is hybrid search and why use it?
A:  Hybrid search combines vector (semantic) search with keyword (BM25/regex)
     search. Vector search finds conceptually similar content even with
     different words. Keyword search finds exact term matches. Together
     they improve recall — vector might miss exact product names, keyword
     might miss paraphrased questions.

Q12: How did you structure the backend code?
A:  Clean architecture with 4 layers:
     Endpoints (thin HTTP handlers) → Services (business logic) →
     Repositories (MongoDB access) → Schemas (Pydantic DTOs).
     AI layer is separate (ai/ package). Cross-cutting concerns in
     middleware/ and core/. Dependency injection via FastAPI Depends().

Q13: What design patterns did you use?
A:  Repository Pattern (data access), Service Layer (business logic),
     Dependency Injection (FastAPI Depends), Strategy Pattern (LLM/vector
     fallbacks), Factory Pattern (get_service closures), Middleware Pipeline
     (audit, CORS, rate limit), Async Task Queue (Celery).

Q14: How does the website crawler work?
A:  User submits URL → Celery job starts → WebsiteCrawler class uses
     BeautifulSoup + httpx async client → crawls same-domain links up to
     depth 2, max 50 pages → each page's text fed into KnowledgeService.ingest_text()
     → same ingestion pipeline as manual uploads. Re-syncs daily via Celery Beat.

Q15: What is confidence scoring?
A:  Two signals: (1) average cosine similarity of retrieved chunks (retrieval
     quality), (2) LLM self-assessment asking "rate your confidence 0-1".
     Final score = max of both. Low confidence triggers escalation to prevent
     the bot from giving wrong answers to customers.

Q16: How is the frontend state managed?
A:  Three layers: (1) Zustand for auth state (user, workspace, tokens) —
     persisted to localStorage. (2) TanStack React Query for all API/server
     state — handles caching, refetching, loading states. (3) Local component
     state (useState) for form inputs and UI toggles.

Q17: What are the subscription plans?
A:  Free: 10 docs, 1K messages, 1 chatbot, 2 users
     Starter ($29): 100 docs, 10K messages, 3 chatbots, 5 users
     Professional ($99): 1000 docs, 100K messages, 10 chatbots, 20 users
     Enterprise (custom): unlimited everything + SSO + dedicated support
     (Stripe billing integration is stubbed — plans defined in BillingService)

Q18: What is Celery and why did you use it?
A:  Celery is a distributed task queue. Document ingestion can take 30+
     seconds (extract, chunk, embed 100s of chunks). Without Celery, the
     API would block until complete. With Celery, API returns immediately,
     processing happens in background worker. Redis is the message broker.

Q19: How would you scale this to thousands of companies?
A:  (1) Horizontal scale: multiple FastAPI instances behind Nginx load balancer
     (2) MongoDB Atlas with sharding by workspace_id
     (3) Pinecone serverless handles vector scaling automatically
     (4) Scale celery-worker replicas for ingestion throughput
     (5) Redis Cluster for HA caching
     (6) S3 for unlimited document storage
     (7) CDN for widget.js delivery

Q20: What are the limitations / what would you improve?
A:  Honestly: Stripe billing not wired, voice/WhatsApp/Slack are stubs,
     API key auth middleware not connected to routes, no CSRF protection,
     widget missing file upload and dark mode, semantic search doesn't cover
     conversations, notifications collection unused, no SSO, no document
     versioning, no team collaboration on knowledge base approval workflow.

Q21: What is Pinecone and why not use MongoDB for vectors?
A:  Pinecone is a purpose-built vector database optimized for similarity
     search in high-dimensional spaces (1536 dims). MongoDB has Atlas Vector
     Search but Pinecone offers better performance at scale, namespace
     isolation, metadata filtering, and managed infrastructure. We have
     an in-memory fallback for development without Pinecone API key.

Q22: Explain the chunking strategy.
A:  We use LangChain RecursiveCharacterTextSplitter with chunk_size=800
     characters and chunk_overlap=150. It tries to split on paragraph
     boundaries first, then sentences, then words — preserving semantic
     coherence. Overlap ensures context isn't lost at chunk boundaries.
     800 chars ≈ 500-1000 tokens, within the recommended RAG range.

Q23: What is the embeddable widget and how does it integrate?
A:  A standalone JavaScript bundle built with Vite (IIFE format).
     Customer adds one script tag to their website:
     <script src="widget.js" data-chatbot-id="xxx"></script>
     Widget creates a floating chat button, fetches chatbot config from
     public API, manages conversation via localStorage, sends messages
     to public chat endpoint, shows typing indicator and feedback buttons.

Q24: How do you prevent AI hallucinations?
A:  Multiple techniques:
     (1) RAG — answer grounded in retrieved context only
     (2) Prompt rules — "answer ONLY from context, say I don't know otherwise"
     (3) Confidence scoring — low confidence triggers human escalation
     (4) Source citations — customer can verify answer against sources
     (5) Temperature 0.3 — lower temperature = more deterministic responses

Q25: What monitoring is in place?
A:  Prometheus scrapes /metrics endpoint from FastAPI (via prometheus-fastapi-
     instrumentator). Grafana dashboards on port 3001. Audit logs in MongoDB
     for all mutations. Health check endpoint at /health. Celery task retries
     with logging on failure.
